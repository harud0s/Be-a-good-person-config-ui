# QA Subagent 審核機制與開發規範

## 核心規則
每次有**需要改動專案的任務**，必須遵守以下流程：
1. **問題測驗與計畫制定**：執行 Pre-task 7 問題測驗，並提交詳細計畫。
2. **QA 審核循環**：
   - 全部的 subagent 第一條訊息必須是：「閱讀 \docs 和現在的專案程式碼」
   - 將計畫交給一個 QA subagent 審核，並根據結果修改計畫。
   - **必須要有連續 3 位 QA 審核通過**，才可以進入實作階段。
   - 如果審核不通過，必須重新制定並提交計畫，然後重新由 3 位 QA 審核。
3. **每次 subagent 回覆後**：必須來查看此規則紀錄以確定下一步。
4. **成果審核 (Post-execution)**：
   - 實作完成後，直接請 3 位 QA 進行最終成果審核。
   - 統整 3 位 QA 的審核結果成一份審核報告給使用者。
   - 若通過：執行 `git push`。
   - 若不通過：重新制定並提交計畫。

## 系統規範與限制
- **無 JSON 汙染 (No JSON Pollution)**：不允許因為 UI 功能 (例如拖曳、邊界標示) 在輸出的 JSON 中新增額外的 key (例如不可新增 `isBoundary` 儲存至檔案中)。
- **邊界改變邏輯 (Boundary Logic)**：對於 `event_sequence`，拖動分界線時，分界線會「跨越」事件，被跨越的事件其內部的 `year` 和 `semester` 數值會自動被修改為分界線另一邊的學期。
- **純文字模式**：提供 JSON 格式驗證，支援 Tab 縮排，允許強制儲存有驗證警告但格式合法的 JSON。
